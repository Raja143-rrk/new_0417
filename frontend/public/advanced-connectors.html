Advanced Connectors

Experimental and custom connectors live here until they are production-ready.

Experimental Relational

These engines are planned for adapter-based implementation, but they are not part of the supported production migration flow yet.

Oracle
Planned
Needs dedicated metadata, DDL extraction, PK lookup, preflight validation, and routine conversion support.
DB2
Planned
Requires adapter implementation for schema discovery, object extraction, and engine-specific data migration SQL.
Teradata
Planned
Needs connector, metadata APIs, type mapping, and target-aware transformation logic.

Cloud And Non-Standard

These connectors either need partial-specialized support or a separate migration model from the current relational engine.

Amazon Redshift
Partial
Close to PostgreSQL, but should be stabilized as a separate adapter before production use.
Azure SQL
Partial
Close to SQL Server, but should be exposed only after adapter-specific testing and validation rules.
Google Cloud SQL
Partial
Depends on whether the instance is MySQL or PostgreSQL; production support should be split by backing engine.
MongoDB
Separate Module
Document databases need a different extraction and migration model than the current relational DDL engine.

Recommended Product Structure

Use the main migration page for production-supported engines only: MySQL, PostgreSQL, SQL Server, and Snowflake. Keep this advanced module for planned or experimental connectors until each adapter reaches the same quality bar.
